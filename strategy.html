Synapse is a fully autonomous equity trading system that orchestrates five LLMs, a deterministic technical analysis engine, and a multi-gate veto chain to generate, validate, and execute trades on Alpaca. Every component — from schema validation to bracket order execution — was architected and implemented by a single developer.

Each ticker in the universe is processed sequentially through all 6 steps per 30-minute cycle. Any gate failure terminates processing for that ticker immediately — the system does not attempt to recover or override. Direction is determined by weighted consensus vote across all four signal models: Grok (confidence×1.0), Claude (confidence×0.8), Gemini (regime-asymmetric: risk_on +2 long, risk_off +3 short), and Perplexity (news ±2 pts). The side with the higher score wins, but must exceed a minimum margin of 2.0 points to qualify. This replaces the original Grok-monopoly design, enabling short entries based on fundamental or macro conviction and eliminating structural long-bias.

Model Responsibilities & Integration
Step 1 — Grok 4 SIGNAL

Role: Social sentiment gate via live X/Twitter search. Scans posts and web news for momentum, narrative shifts, and unusual activity per ticker using xAI's /v1/responses endpoint with x_search + web_search tools.

Output schema: direction (long/short/flat), confidence (1–10), momentum_state (accelerating/decelerating/stable), narrative string, unusual_activity boolean.

Gate logic: Flat signal → ticker skipped entirely. Confidence < 6 → vetoed at Aggregator (Rule 6). Direction feeds into the consensus vote (weighted by confidence) alongside Claude, Gemini, and Perplexity.

Step 2 — Perplexity Sonar GATE

Role: Real-time news verification. Cross-checks Grok's sentiment signal against live web data — SEC filings, earnings, analyst actions, corporate events within the last 24 hours.

Output schema: material_news boolean, news_direction enum, news_summary, grok_signal_verified boolean, trading_halt_risk boolean, sources array.

Step 3a — Claude Sonnet 4.6 GATE

Role: Fundamental analysis gate. Evaluates earnings quality, balance sheet risk, accounting red flags, and competitive position. Returns a confidence score and position size multiplier (0.5–1.5×) that directly scales order size.

Gate logic: fundamental_view = "avoid" → hard veto (Rule 3). hold_duration = "avoid" → hard veto (Rule 4). Confidence < 6 → veto (Rule 7).

Cost optimization: Switched from Opus 4.5 ($15/$75 per 1M) to Sonnet 4.6 ($3/$15 per 1M) — ~80% reduction with comparable gate quality on structured extraction tasks.

Step 3b — Gemini 2.5 Flash GATE

Role: Macro regime classifier. Determines market-wide risk regime (risk_on / neutral / risk_off) and returns sector avoidance list, maximum exposure recommendation, and position size scalar.

Gate logic: macro_veto = true → full cycle veto (all tickers blocked). Response is cached intraday — Gemini is called once per cycle, not per ticker.

Cost optimization: Cheapest model in the pipeline ($0.15/$0.60 per 1M). Single call amortized across all tickers makes per-ticker cost effectively zero.

Step 4 — Aggregator: Consensus Vote + Veto Chain

The Aggregator operates in two phases. Phase 1 — Direction consensus: A weighted vote across all four models determines trade direction. Grok contributes confidence×1.0, Claude confidence×0.8, Gemini regime (risk_on +2 long, risk_off +3 short — asymmetric because markets drop faster), Perplexity news ±2 pts. The side with the higher score wins, but must exceed a minimum margin of 2.0 points to qualify as consensus. Phase 2 — Veto chain: 7 binary rules evaluated in order. The first failure terminates with NO_TRADE.

Position sizing formula:

# Position size = base × claude_mult × gemini_scalar × momentum_scalar × news_penalty × equity size_usd = (0.10 * claude.multiplier * gemini.scalar * momentum_scalar * news_penalty * portfolio_value) # Effective range: 0.3% – 15% of equity # claude.multiplier ∈ [0.5, 1.5] # gemini.scalar ∈ [0.25, 1.0] # momentum_scalar ∈ [0.25, 1.0] (SPY/QQQ EMA-9 on 30m bars) # news_penalty = 0.85 if no material news, else 1.0 size_usd = min(size_usd, 0.07 * portfolio_value) # hard cap (7% equity)

Technical Analysis Engine
Step 5 — TechLevels: Deterministic Structure Detection PYTHON

No LLM involvement. Pure Python/NumPy/Pandas computation on OHLCV bars fetched from Alpaca. The engine runs 5 independent detectors on two timeframes (daily 12-month + hourly 30-day), scores detected levels, and snaps SL/TP to the nearest structure within ATR-bounded ranges.

  • Left window: 5 bars
  • Right window: 3 bars

  • Touch count tracking

  • 50 price bins across range
  • Volume-at-price bucketing

  • Identifies accumulation zones
  • Volume-weighted strength

Detector 3: Liquidity Sweeps

  • Equal lows/highs clustering
  • 0.1% band tolerance

  • Retail stop accumulation

Detector 4: Key Moving Averages

  • 20 / 50 / 200 EMA on daily bars
  • Dynamic support/resistance
  • Confluence with static levels

Detector 5: VWAP

  • Volume-weighted average price
  • Intraday anchor level
  • Institutional reference point
Breach Detection

Each detected level is tested for historical breaches — bars that closed through the level beyond a 0.1% tolerance band. Each breach penalises the level's strength score by 0.15 (capped at −0.45). This identifies damaged levels that held visually but have been broken through, including polarity flips (old support acting as resistance).

for lvl in levels: band = lvl.price * 0.001 if lvl.sr == 'support': breach_count = np.sum(close < (lvl.price - band)) else: breach_count = np.sum(close > (lvl.price + band)) lvl.breaches = max(lvl.breaches, breach_count) # Penalty: -0.15 per breach, max -0.45 score -= min(lvl.breaches * 0.15, 0.45)
SL/TP Snap Logic

After scoring all levels, the engine snaps the ATR-default SL/TP to the nearest qualifying structure level. Snap must stay within ATR-bounded ranges to prevent absurd risk profiles.

Order Execution & Risk Management
Step 6 — GPT-4o Order Construction

Receives structure-adjusted SL/TP from the TechLevels engine (or ATR defaults as fallback). Constructs a market order with quantity, stop loss, and take profit. The prompt explicitly instructs GPT-4o to USE the pre-computed levels, not recalculate them. No limit price is computed — market orders fill at best available price, eliminating the spread-crossing fiction.

Bracket order execution: Orders are submitted to Alpaca as OCO (One-Cancels-Other) bracket orders. The parent is a market buy/sell; the two legs are stop-loss and take-profit orders. Alpaca manages exit execution autonomously. After parent cancellation, child legs are explicitly verified and orphans force-cancelled.

Position Sentinel — Active Risk Monitoring
Architecture

The system runs two phases per cycle. Phase A (Position Monitor) runs on every cycle regardless of whether the daily trade cap has been hit. It checks Grok sentiment and Perplexity news on every open position, plus the already-cached Gemini macro regime. If any sentinel trigger fires, the position is closed immediately via market order. When no sentinel trigger fires, the system checks trailing stop conditions: at +1× ATR gain, the stop moves to breakeven (entry price); at +2× ATR gain, the stop trails at 1× ATR below the current price. Stops only tighten — never widen.

Phase B (Entry Pipeline) is the 6-step pipeline described above. It only runs when the daily trade cap has not been reached.

This means the system is never blind to breaking events. Even after 5 trades are filled and no more entries are allowed, the sentinel continues monitoring every 30 minutes. A Fed surprise, tariff announcement, or halt risk detected by Perplexity will trigger an early exit before the mechanical SL is hit.

Index Technical Pre-Filter

Before any individual stock enters the pipeline, SPY and QQQ are checked for short-term technical regime. This is a deterministic check — zero API cost — that runs once per cycle. Based on academic evidence that ~90% of price discovery starts in index futures/ETFs and propagates to constituents (Hasbrouck 2003), and that correlations spike asymmetrically during selloffs (Ang & Chen 2002).

Four signals scored per index: (1) Price vs hourly EMA-20 (trend), (2) 5-bar hourly momentum (direction), (3) Price vs daily SMA-50 (intermediate trend), (4) Last bar wick analysis (resistance rejection / support bounce). Scores averaged across SPY + QQQ.

Aggregate score ≤ -1.0Bearish — block all longs
Aggregate score ≥ +1.0Bullish — block all shorts
Score between -1.0 and +1.0Neutral — no filter applied
Entering longs while SPY/QQQ reject resistance is fighting the ETF arbitrage transmission chain. This filter prevents it at zero cost.
Asymmetric Sentinel Sensitivity

The Position Sentinel (Phase A) now carries an additional trigger that exploits the asymmetry of index-to-stock correlation. Ang & Chen (2002) demonstrate that downside correlations are 11.6% higher than normal — when SPY/QQQ sell off, individual stocks are dragged down harder and faster than they are lifted during rallies. ETF arbitrage (Ben-David et al. 2018) mechanically forces constituent prices to converge, regardless of individual stock fundamentals.

Once per sentinel cycle, SPY and QQQ intraday momentum is computed from today's hourly bars (open-to-current change). If a sharp selloff is detected, all open long positions are closed immediately — the system does not wait for Grok or Perplexity to confirm. The contagion is mechanical, not sentiment-driven.

Avg SPY/QQQ intraday ≤ -0.8%Selloff — close all longs
Any single index intraday ≤ -1.2%Sharp selloff — close all longs
Avg SPY/QQQ intraday ≥ +0.8%Rally — close all shorts
Any single index intraday ≥ +1.2%Sharp rally — close all shorts
This is Trigger 5 in the sentinel chain. It fires before the "position OK" conclusion. Zero API cost — Alpaca bar data only.
Index Momentum Position Scalar

Enhances the Gemini macro gate with a deterministic, quantitative position-size adjustment. Before each new entry, SPY and QQQ 30-minute bars are checked against their EMA-9. When both indices trade below EMA-9, the system scales down position size for longs proportional to the intraday decline magnitude. The reverse applies for shorts during rallies.

This does not replace Gemini's LLM-based macro analysis — it multiplies on top of the existing claude_mult × gemini_scalar formula. The final position size becomes: BASE_POSITION_SIZE × claude_mult × gemini_scalar × momentum_scalar × portfolio_value.

Both SPY/QQQ above EMA-9 (30m)1.0× — full size
Mixed signals (one above, one below)0.75× — mild reduction
Both below EMA-9, intraday > -0.3%0.50× — half size
Both below EMA-9, intraday ≤ -0.3%0.50× — half size
Both below EMA-9, intraday ≤ -0.6%0.25× — quarter size
Entering full-size longs against index momentum is empirically suboptimal (Hasbrouck 2003). This scalar prevents it without blocking the trade entirely. Zero API cost.
Ticker Universe — Selection Criteria

11 tickers selected on three filters. Social signal density: Grok scans X/Twitter for direction — tickers with low mention volume produce noisy, low-confidence signals. Every ticker in the universe has consistently high daily social volume (NVDA, TSLA, META, PLTR, HOOD lead retail/fintwit; AAPL, MSFT, AMZN cover institutional + retail; JPM covers financials).

Liquidity and spread: All 11 trade >10M shares/day with sub-penny effective spreads. This matters for bracket order fills and fast sentinel closes — illiquid names would slip through stops or fail to fill on early exits.

Index coverage: QQQ and SPY serve as macro regime proxies. They let the system trade directional macro views when Grok detects strong index-level sentiment (post-FOMC momentum, tariff reaction) and provide natural hedging against single-name positions.

Why not more? Each ticker costs 4–5 API calls per cycle. At 11 tickers × ~5 calls × 48 cycles/day, that's ~2,640 API calls daily. Adding tickers scales cost linearly. Why no small/mid caps? Low social volume → unreliable Grok signals. Low liquidity → poor fills. The veto chain already kills ~70% of signals — low-quality tickers just burn API spend on trades that get vetoed.

Design Decisions

Sequential over parallel pipeline. Each model's output feeds the next. Claude and Gemini run in parallel (Step 3) because they're independent gates, but Perplexity must pass before either is invoked. This isn't a limitation — it's a cost optimization that prevents wasting expensive Claude calls on tickers that will be vetoed by cheap Perplexity checks.

Pydantic schema enforcement on every LLM call. Every model response is parsed into a typed Pydantic model with field constraints (ge=, le=, min_length=). Parse failures produce NO_TRADE, never fallback values. The system does not guess what the LLM meant.

Deterministic computation where LLMs aren't needed. Python handles all technical analysis math (pivot detection, volume clustering, breach counting, SL/TP snapping). The Grok chart critic — which re-evaluated these same levels via LLM — was removed after finding it added 22 seconds of latency and zero new information over the deterministic output.

No database. Trade log is a JSON file. State is a JSON file. Dashboard data is a JS file. This is a single-user paper trading system — the complexity of a database would be pure overhead. File I/O is atomic enough for a single-threaded async loop.

Concurrency Model

Main loop is async (asyncio). LLM calls within a ticker are sequential (output feeds next step). Claude + Gemini are gathered in parallel via asyncio.gather(). Background thread polls Alpaca every 60s for live P/L and trade log reconciliation. No multiprocessing — everything runs in one process.

Cost Optimization Decisions

Perplexity as pre-gate (Step 2): At $1/$1 per 1M tokens, Perplexity is the cheapest model. Placing it before Claude ($3/$15) means doomed tickers are killed before the expensive call.

Gemini caching: One call per cycle, not per ticker. With 11 tickers, this is an 11× cost reduction on macro analysis.

Claude model switch: Opus 4.5 → Sonnet 4.6 reduced Claude's per-call cost by ~80% ($0.018 → $0.004 avg). Gate accuracy remained comparable — structured extraction with Pydantic schemas doesn't require frontier reasoning.

TechLevels (Step 5): Zero API cost. All computation is local Python. This is the most complex analysis step in the pipeline and it costs nothing per call.

Trade Log Reconciliation

Every 60 seconds, the background thread fetches live positions from Alpaca and cross-references against the trade log. Trades whose ticker is no longer present in Alpaca's open positions are marked as closed. Exit price is sourced from Alpaca's closed orders API when available, falling back to the stop-loss price (conservative estimate) when the API doesn't return a fill. P/L is computed and persisted to the trade log, which the dashboard reads on next refresh.

What This Project Demonstrates

Multi-API orchestration. Five different LLM providers, each with different auth patterns, response formats, and failure modes, coordinated in a single pipeline with consistent error handling.

Hybrid AI architecture. Knowing when to use deterministic computation vs. LLM inference, and designing the boundary between them (Python detects levels, LLM validates judgment).

Production-grade risk engineering. Kill switches, drawdown halts, exposure caps, per-ticker cooldowns, parse failure handling. The system is designed to fail safely — every unknown state maps to NO_TRADE.

Cost-conscious LLM deployment. Pre-gate ordering, model selection (Sonnet over Opus), caching strategies, and zero-cost local computation where LLMs aren't needed.

Schema-driven development. Pydantic models define the contract between every system boundary. No untyped dicts, no string parsing, no "hope the LLM returns the right format."

Full-stack ownership. Backend pipeline, technical analysis engine, order execution, risk management, dashboard, cost tracking, and deployment — single developer, end to end.
